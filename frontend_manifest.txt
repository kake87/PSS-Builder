# Требования и манифест фронтенда

## Проект: Конструктор архитектур систем безопасности

---

# 1. Назначение фронтенда

Фронтенд предназначен для инженерной сборки, просмотра, проверки и корректировки архитектур систем безопасности. Это не маркетинговый интерфейс и не массовый пользовательский продукт. Это рабочий инженерный инструмент, ориентированный на быстрый переход от идеи к формализованной схеме проекта.

Фронтенд должен обеспечивать две основные модели работы:

1. **Графический режим** — пользователь вручную собирает архитектуру из устройств, портов и линий.
2. **AI-режим** — пользователь формулирует задачу текстом, а система создает и отображает черновую архитектуру, которую затем можно редактировать вручную.

Главная ценность фронтенда — не красота схемы, а инженерная управляемость модели.

---

# 2. Принципы фронтенда

## 2.1 Frontend is not the source of truth

Фронтенд не является источником истины. Он отображает и редактирует проектную модель, которая хранится в Project Engine.

## 2.2 Diagram follows data

Схема должна быть следствием данных, а не самостоятельной сущностью.

## 2.3 Ports are first-class citizens

Подключение осуществляется порт-к-порту, а не устройство-к-устройству.

## 2.4 Validation is visible

Ошибки, предупреждения и ограничения должны быть видны прямо на схеме и в боковых панелях.

## 2.5 AI assists, not overrides

AI-помощник предлагает изменения, но не должен без контроля пользователя разрушать инженерную структуру.

## 2.6 Fast engineering workflow first

Скорость работы инженера важнее декоративности интерфейса.

---

# 3. Цели фронтенда

Фронтенд должен позволять:

* собрать архитектуру вручную за 3–10 минут для типового объекта;
* принять черновик от AI и быстро довести его до инженерно приемлемого состояния;
* видеть структуру портов, линий, типов кабеля, длин и ограничений;
* получать немедленную визуальную обратную связь от валидатора;
* подготавливать проект к переходу в спецификацию, расчет и текстовое описание.

---

# 4. Роли пользователя

На первом этапе достаточно одной основной роли.

## 4.1 Инженер-проектировщик

Может:

* создавать и редактировать проекты;
* добавлять оборудование;
* соединять порты;
* задавать длины и параметры линий;
* запускать AI-помощника;
* видеть ошибки и рекомендации;
* подготавливать проект к выгрузке.

В дальнейшем могут быть добавлены:

* reviewer;
* sales engineer;
* admin каталога.

---

# 5. Основные сценарии использования

## 5.1 Ручная сборка

1. Пользователь создает проект.
2. Открывает библиотеку оборудования.
3. Перетаскивает устройства на canvas.
4. Соединяет порты.
5. Указывает тип кабеля и длину.
6. Получает статусы валидации.
7. Корректирует архитектуру.

## 5.2 AI-генерация

1. Пользователь вводит инженерную задачу текстом.
2. Фронтенд отправляет задачу AI-конвейеру.
3. Получает proposal.
4. Показывает пользователю предварительную схему.
5. Подсвечивает предположения и неопределенности.
6. Дает возможность применить, отклонить или частично принять изменения.

## 5.3 Анализ существующей схемы

1. Пользователь открывает проект.
2. Видит ошибки, предупреждения, загрузку портов, PoE, длины линий.
3. Переходит к проблемным участкам через UI.

---

# 6. Состав фронтенда

Фронтенд состоит из следующих функциональных областей:

1. **Project Shell**
2. **Canvas / Node Editor**
3. **Equipment Library**
4. **Properties Panel**
5. **Validation Panel**
6. **AI Assistant Panel**
7. **Topology / Zone Navigation**
8. **Toolbar / Commands**
9. **Project Metadata View**

---

# 7. Общая компоновка интерфейса

Рекомендуемая компоновка:

* **слева** — библиотека оборудования / дерево проекта / зоны;
* **по центру** — canvas схемы;
* **справа** — свойства выбранного элемента / валидация / AI-действия;
* **сверху** — toolbar и глобальные действия;
* **снизу или справа вкладкой** — список ошибок, журнал действий, события AI.

---

# 8. Canvas / Node Editor

## 8.1 Назначение

Canvas — основной рабочий слой. Он должен позволять редактировать инженерный граф проекта.

## 8.2 Объекты на canvas

На canvas должны отображаться:

* устройства;
* порты;
* линии связи;
* зоны/группы;
* индикаторы состояния;
* подписи длин и типов кабеля.

## 8.3 Требования к canvas

* pan / zoom;
* drag-and-drop узлов;
* multi-select;
* выделение рамкой;
* snap-to-grid опционально;
* auto-layout опционально;
* сохранение позиций узлов;
* быстрые команды через контекстное меню.

## 8.4 Требования к производительности

* комфортная работа минимум до 300–500 узлов без деградации UX;
* минимизация лишних перерисовок;
* виртуализация боковых списков;
* асинхронная загрузка тяжелых данных.

---

# 9. Узлы устройств

## 9.1 Общая модель отображения

Каждый узел устройства должен показывать:

* тип устройства;
* краткую модель;
* ключевой инженерный статус;
* порты;
* локальные ошибки/предупреждения.

## 9.2 Визуальные требования

Устройство должно быть читаемым на масштабе 70–100% и оставаться узнаваемым на меньшем масштабе.

## 9.3 Режимы детализации

Нужно минимум три уровня детализации:

1. collapsed — компактная карточка;
2. normal — карточка с портами;
3. detailed — карточка с расширенной инженерной информацией.

## 9.4 Индикация состояния

Узел должен явно показывать:

* error;
* warning;
* valid;
* draft/unresolved;
* selected;
* locked (если появится позже).

---

# 10. Порты и точки подключения

## 10.1 Порты как обязательная сущность UI

Порты должны быть визуально отображены как точки подключения.

## 10.2 Типы портов

На первом этапе поддержать:

* ethernet RJ45;
* SFP / fiber;
* power;
* RS-485;
* relay;
* alarm in/out;
* Wiegand.

## 10.3 Направление порта

UI должен уметь показывать:

* bidirectional;
* input;
* output.

## 10.4 Поведение соединения

Пользователь соединяет порт с портом. Если соединение несовместимо, UI либо блокирует действие, либо разрешает его с явным error-state после проверки.

## 10.5 Визуальные правила

* Ethernet и fiber должны визуально отличаться.
* Input/output порты должны иметь понятную иконографику.
* Занятый порт должен быть заметен.
* Свободные порты должны быть различимы.

---

# 11. Линии связи

## 11.1 Что отображает линия

Каждая линия должна уметь показывать:

* тип кабеля;
* длину;
* состояние валидации;
* направление там, где это применимо;
* принадлежность к uplink / access / control / power.

## 11.2 Статусы линии

* ok;
* near limit;
* warning;
* error;
* unresolved.

## 11.3 Минимальный набор редактируемых параметров

* cable type;
* route length;
* vertical length;
* reserve;
* service loop;
* environment;
* installation type.

## 11.4 Визуальная индикация

Линия должна менять внешний вид по severity, но без визуального перегруза.

---

# 12. Зоны и группировка

## 12.1 Назначение

Зоны нужны для логического разбиения архитектуры.

## 12.2 Примеры зон

* Server room;
* Perimeter North;
* Perimeter South;
* Checkpoint 1;
* Warehouse interior.

## 12.3 Возможности

* создание зоны;
* привязка устройств к зоне;
* сворачивание/разворачивание зоны;
* фильтрация по зонам;
* навигация по зоне.

---

# 13. Библиотека оборудования

## 13.1 Назначение

Панель оборудования должна предоставлять доступ к рабочему каталогу устройств.

## 13.2 Функции

* поиск по модели, вендору, категории;
* фильтрация по ролям;
* фильтрация по capabilities;
* избранные и preferred устройства;
* быстрый drag-and-drop на canvas.

## 13.3 Минимальные категории MVP

* camera;
* switch;
* server;
* controller;
* terminal;
* cable;
* cabinet;
* UPS.

## 13.4 Инженерная подача

Карточка устройства в библиотеке должна сразу показывать полезные параметры, а не только название.

---

# 14. Панель свойств

## 14.1 Назначение

Правая панель должна быть главным местом редактирования выбранного объекта.

## 14.2 Для устройства

Показывать:

* общие данные;
* спецификации;
* роли;
* capabilities;
* порты;
* локальные ошибки;
* статус в проекте.

## 14.3 Для линии

Показывать:

* тип кабеля;
* длины;
* рассчитанную длину;
* среду прокладки;
* installation type;
* предупреждения и нормативные замечания.

## 14.4 Для зоны

Показывать:

* имя;
* список устройств;
* локальную статистику;
* ошибки зоны.

---

# 15. Validation UI

## 15.1 Основная идея

Валидация должна быть видна в трех местах:

1. локально на узлах и линиях;
2. в панели проблем;
3. в сводке проекта.

## 15.2 Панель проблем

Панель должна поддерживать:

* группировку по severity;
* группировку по типу проблемы;
* фильтр по зоне;
* переход к объекту по клику;
* показ рекомендации.

## 15.3 Типы проблем MVP

* incompatible interfaces;
* exceeded port capacity;
* exceeded PoE budget;
* cable length over limit;
* near cable limit;
* missing uplink;
* missing required component.

## 15.4 Поведение

Валидация должна обновляться быстро и быть понятной. Пользователь должен понимать не только что плохо, но и что делать дальше.

---

# 16. AI Assistant UI

## 16.1 Роль AI в интерфейсе

AI — это помощник по генерации и модификации схемы.

## 16.2 Основные режимы

* generate from prompt;
* modify selected zone;
* suggest correction;
* explain validation issue;
* propose topology alternative.

## 16.3 UX-принципы

* AI не должен скрыто менять проект;
* каждое действие AI должно оформляться как proposal;
* пользователь должен видеть diff: что будет добавлено, удалено, изменено;
* пользователь может принять все, принять частично, отклонить.

## 16.4 История действий AI

Нужен журнал AI-действий:

* prompt;
* proposal summary;
* applied/rejected state;
* timestamp.

# 17. Команды и взаимодействия

## 17.1 Базовые команды

* add device;
* duplicate device;
* connect ports;
* remove link;
* assign zone;
* edit link properties;
* run validation;
* invoke AI;
* undo;
* redo.

## 17.2 Горячие клавиши

На MVP желательно поддержать:

* delete;

---

# 18. Состояния данных

Фронтенд должен различать:

* persisted state;
* local editing state;
* AI proposal state;
* validation state;
* loading state;
* error state.

Нельзя смешивать эти состояния в один неразличимый store.

---

# 19. Архитектурные требования к состоянию фронтенда

## 19.1 Общие требования

Фронтенд должен иметь предсказуемую state-модель.

## 19.2 Рекомендуемое разделение

* server state: данные проекта, каталога, validation results;
* client UI state: выделение, панели, zoom, filters;
* transient editing state: черновые редактирования;
* AI proposal state: предложенные изменения.

## 19.3 Принцип

Нельзя хранить бизнес-логику только в компонентах canvas.

---

# 20. Требования к сетевому взаимодействию

Фронтенд должен работать через API и не дублировать инженерную логику в разных местах.

Минимально нужны endpoint-и/операции:

* get project;
* update node position;
* add device;
* remove device;
* connect ports;
* update link;
* run validation;
* request AI proposal;
* apply proposal;
* get equipment catalog.

---

# 21. Нефункциональные требования

## 21.1 Производительность

* первый meaningful render должен быть быстрым;
* операции drag/drop и pan/zoom должны быть плавными;
* обновление валидации не должно стопорить canvas;
* тяжелые панели должны обновляться независимо.

## 21.2 Надежность

* UI не должен терять локальные изменения при временной ошибке сети;
* ошибки API должны отображаться внятно;
* применение AI proposal должно быть атомарным.

## 21.3 Расширяемость

Архитектура фронтенда должна позволять добавлять:

* новые типы устройств;
* новые порты;
* новые validation rules;
* дополнительные представления (таблица, BOM, topology view).

## 21.4 Поддерживаемость

* явное разделение domain/components/features;
* минимум скрытой магии;
* типобезопасность;
* изолированное тестирование логики UI.

---

# 22. MVP фронтенда

В MVP обязательно должны быть:

1. Project shell
2. Canvas c node-based редактором
3. Устройства: camera, switch, server
4. Видимые порты
5. Соединение порт-к-порту
6. Редактирование типа кабеля и длины
7. Базовая validation panel
8. Библиотека оборудования
9. AI prompt → proposal → apply
10. Undo / redo

В MVP не обязательны:

* многоуровневые сценарии совместной работы;
* продвинутый auto-layout;
* сложные коммерческие модули;
* 3D/plan overlays;
* детальный BOM UI.

---

# 23. Рекомендуемый технологический стек фронтенда

## 23.1 Базовый стек

* React
* TypeScript
* React Flow для node-editor слоя
* TanStack Query для server state
* Zustand или аналог для UI state
* zod для runtime validation клиентских контрактов
* Tailwind или аналог для быстрой UI-сборки

## 23.2 Причины выбора

Этот стек дает:

* быстрый старт;
* хорошую управляемость;
* удобную работу с node-based UI;
* низкий порог итераций.

---

# 24. Архитектура фронтенд-кода

Рекомендуемая структура:

* app/
* pages/
* widgets/
* features/
* entities/
* shared/
* processes/

или другой близкий модульный подход.

Принцип:

* canvas не должен знать детали API;
* catalog UI не должен знать детали AI;
* AI panel не должна напрямую мутировать graph state вне управляемого потока.

---

# 25. Ключевые пользовательские требования

Фронтенд считается успешным, если инженер может:

* быстро начать проект;
* быстро добавить типовые узлы;
* понять, что куда подключено;
* сразу увидеть проблемные линии и перегрузки;
* получить черновик от AI и не потерять контроль над проектом.

---

# 26. Критерии приемки MVP фронтенда

MVP можно считать годным, если он позволяет:

1. создать новый проект;
2. перетащить устройства на canvas;
3. соединить порты;
4. указать длину и тип кабеля;
5. увидеть ошибку превышения длины медной линии;
6. увидеть нехватку портов или PoE;
7. получить AI-черновик по тексту;
8. принять предложенные AI-изменения;
9. сохранить и заново открыть проект без потери структуры.

---

# 27. Манифест фронтенда

Мы строим не просто интерфейс для рисования схем.

Мы строим инженерный рабочий слой, в котором:

* схема подчиняется данным;
* подключение подчиняется портам;
* линия подчиняется физике и нормативам;
* AI подчиняется проектной модели;
* пользователь сохраняет контроль над архитектурой.

Фронтенд должен быть быстрым, читаемым, инженерно честным и устойчивым к росту сложности.

Если интерфейс красив, но не помогает обнаружить ошибку — он провалился.
Если интерфейс помогает собрать рабочую архитектуру быстрее и безопаснее — он выполняет свою задачу.
