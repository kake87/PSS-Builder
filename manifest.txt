Технический манифест
Конструктор архитектур систем безопасности
1. Назначение системы
Что это такое в реальном мире и зачем это придумали

Инженеры систем безопасности (видеонаблюдение, СКУД, периметральная охрана) постоянно выполняют одну и ту же задачу: собирают архитектуру системы из камер, серверов, коммутаторов, контроллеров доступа и линий связи.

Этот процесс обычно выполняется вручную:

инженер держит архитектуру в голове

проверяет совместимость оборудования

оценивает расстояния кабелей

проверяет нормативы

рассчитывает емкость сети и хранения

формирует спецификацию

Это медленный и ошибкоопасный процесс.

Цель проекта — создать инженерный конструктор архитектур систем безопасности, который позволяет:

быстро собирать архитектуру систем

проверять техническую корректность

учитывать реальные ограничения

автоматически формировать черновой проект

Система предназначена для внутреннего использования инженером, а не как массовый продукт.

2. Основная концепция системы

Система представляет архитектуру безопасности как граф инженерных объектов.

Device ── Port ── Link ── Port ── Device

Основные сущности:

Device        (устройство)
Port          (порт устройства)
Link          (линия связи)
Cable         (тип кабеля)
Rule          (нормативное правило)
Project       (проект системы)
Template      (шаблон решения)

Архитектура проекта — это граф устройств и соединений.

Часть 2. ИИ-ядро проекта
Глава 1. Как агент вообще должен работать
Раздел 1. Архитектура ИИ-агента: не “один умный бот”, а конвейер из ролей

В реальной жизни почти все провальные AI-инструменты для инженерии ломаются об одну и ту же вещь: им дают большой промпт, и они пытаются сразу “придумать проект”. На выходе получается текст, который звучит уверенно, но инженерно плавает.

Тебе такой вариант не нужен.

Тебе нужен агент не как “болтун”, а как управляемый технологический процесс. То есть не один универсальный мозг, а несколько шагов, где каждый делает свою узкую работу, а результат передаётся дальше.

Простым языком: агент должен работать как бригада, а не как один “суперсотрудник”.

Пример из жизни.
Когда ты сам собираешь решение, ты ведь не думаешь одним слитным куском:

сначала понимаешь задачу,

потом выделяешь сущности,

потом прикидываешь типовую схему,

потом подбираешь железо,

потом проверяешь связи и ограничения,

потом приводишь это в понятный вид.

Вот агент должен делать то же самое.

Правильная идея выглядит так:

Входной текст
   │
   v
[1] Разбор задачи
   │
   v
[2] Нормализация требований
   │
   v
[3] Выбор шаблона архитектуры
   │
   v
[4] Подбор оборудования из БД
   │
   v
[5] Построение графа устройств и связей
   │
   v
[6] Прогон через валидатор
   │
   v
[7] Формирование результата

Вот это и есть архитектура агента.

Почему это нужно в реальной жизни

Потому что если пропустить промежуточные шаги, агент начнёт:

— выдумывать недостающее,
— путать требования с предположениями,
— подбирать несовместимое оборудование,
— игнорировать длины линий и порты,
— выдавать текст без инженерной структуры.

А если у тебя есть этапы, ты можешь контролировать, где именно ошибка.

Например:

— ошибка на этапе разбора текста,
— ошибка на этапе выбора шаблона,
— ошибка на этапе подбора из БД,
— ошибка на этапе проверки.

Это очень важно.
Потому что управляемая ошибка лучше, чем “магия сломалась где-то внутри”.

Теперь самая важная мысль этого раздела:

Агент не должен генерировать финальный проект напрямую из текста.
Он должен генерировать промежуточные структуры.

То есть не так:

Текст пользователя -> готовый проект

А так:

Текст пользователя
   ->
Структура требований
   ->
Черновая инженерная модель
   ->
Проверенный проект

Это фундамент.

Схема ролей агента

Я бы для твоего проекта разделил агента на 6 логических ролей.

Роль	Что делает	Что отдает дальше
Parser	Вытаскивает смысл из текста	Набор требований
Normalizer	Превращает туманные слова в формальные поля	Структурированный input
Planner	Выбирает типовую топологию	Черновой план решения
Selector	Подбирает устройства из БД	Конкретный набор железа
Topology Builder	Строит граф: устройства, порты, связи	Проектную модель
Validator	Проверяет порты, длины, PoE, нормы	Ошибки, предупреждения, статус

Это не обязательно 6 отдельных процессов или 6 отдельных LLM-вызовов. На старте это могут быть 3–4 шага. Но логически роли должны существовать.

Теперь разберём их по-человечески.

Parser — “что вообще хочет пользователь”

Это первый фильтр.

Пользователь пишет что-то вроде:

“Нужна система для склада, 24 камеры, 2 точки въезда, распознавание номеров, архив 30 дней, часть камер далеко.”

Parser не должен сразу думать про модели камер.
Он должен вытащить смысл:

{
  "object_type": "warehouse",
  "camera_count": 24,
  "entry_points": 2,
  "anpr": true,
  "storage_days": 30,
  "long_distance_segments": true
}

То есть Parser отвечает на вопрос:
“Что было сказано?”

Normalizer — “что это значит в инженерном виде”

Люди пишут не по шаблону.
Например:

— “далеко”
— “небольшой объект”
— “нужен контроль въезда”
— “камеры по периметру”
— “хотим распознавать лица”

Это ещё не инженерные параметры.

Normalizer переводит это в поля системы.

Например:

{
  "site_type": "warehouse",
  "zones": ["perimeter", "indoor", "checkpoint"],
  "analytics": ["anpr"],
  "retention_days": 30,
  "distance_risk": "high",
  "access_control_required": false
}

То есть он отвечает на вопрос:
“Как это записать так, чтобы машина могла с этим работать?”

Planner — “какой тип решения вообще подходит”

Тут агент уже не подбирает модели, а выбирает логику построения.

Например, для склада с длинным периметром Planner может решить:

— нужен шаблон warehouse_perimeter,
— камеры группируются по зонам,
— для удалённых камер нужен уличный шкаф,
— uplink до шкафа лучше делать по оптике,
— ANPR-узлы идут отдельно на точках въезда.

Planner отдаёт не железо, а план архитектуры.

Например:

{
  "template": "warehouse_perimeter",
  "zones": [
    {"name": "perimeter_north", "camera_count": 8},
    {"name": "perimeter_south", "camera_count": 8},
    {"name": "indoor", "camera_count": 4},
    {"name": "entry_points", "anpr_nodes": 2}
  ],
  "topology_intent": [
    "distributed_access_switching",
    "fiber_uplinks_for_remote_zones"
  ]
}

Это очень сильный этап.
Он не даёт агенту сразу прыгать к случайному подбору железа.

Selector — “что именно брать из твоей БД”

Вот здесь агент впервые трогает каталог оборудования.

Но важно: Selector не должен искать “лучшее устройство в мире”.
Он должен выбирать из твоего рабочего набора.

Например:

Задача:
8 уличных камер на периметре
нужен PoE
нужна уличная установка
нужен ночной режим

Selector:
- camera_model = Hikvision_X
- switch_model = PoE_24_ports_Y
- cabinet_model = Outdoor_Cabinet_Z
- uplink = fiber_sm

То есть он отвечает на вопрос:
“Какие реальные позиции из справочника подходят под задуманный план?”

Topology Builder — “собери из этого проектную модель”

Вот здесь начинается самое инженерное.

Builder должен создать не текст, а граф проекта:

— устройства,
— их порты,
— связи между портами,
— типы кабелей,
— длины,
— зоны.

То есть итог не “предлагаю использовать…”, а примерно такая структура:

{
  "devices": [...],
  "ports": [...],
  "links": [...],
  "zones": [...]
}

Именно здесь система становится конструктором, а не чат-ботом.

Validator — “это вообще можно собрать?”

Validator проверяет уже собранный граф.

Например:

— хватает ли портов,
— хватает ли PoE-бюджета,
— не превышена ли длина меди,
— нет ли неверных интерфейсов,
— есть ли uplink между зонами,
— не построена ли логически невозможная схема.

И результат должен быть не “всё хорошо”, а структурированным:

{
  "status": "warning",
  "issues": [
    {
      "type": "link_length",
      "severity": "error",
      "message": "Copper Ethernet segment exceeds allowed calculated length",
      "link_id": "L-204"
    }
  ],
  "recommendations": [
    "Replace uplink with fiber",
    "Move cabinet closer to camera cluster"
  ]
}

Вот это зрелая архитектура агента.

Теперь покажу, как это выглядит целиком.

Теперь очень важная практическая мысль.

Где тут должен быть LLM, а где обычный код?

Это критично. Многие ошибаются и пытаются всё делать через LLM.

Правильнее так:

Этап	Кто делает лучше
Разбор свободного текста	LLM
Нормализация расплывчатых требований	LLM + код
Выбор шаблона	код + правила, иногда LLM
Подбор оборудования из БД	код
Построение графа	код
Валидация	код
Человеческое объяснение результата	LLM

То есть LLM нужен там, где есть человеческая нечеткость.
А где есть точные правила, там должен работать код.

Это ключевая инженерная дисциплина.

Пример плохой архитектуры:

LLM читает запрос
LLM подбирает железо
LLM рисует топологию
LLM решает, что всё валидно

Почему плохо:
LLM не должен быть финальным судьёй по длине линии, PoE-бюджету и портам.

Пример хорошей архитектуры:

LLM понял намерение
Код собрал модель
Код проверил ограничения
LLM объяснил результат человеку

Вот так надо.

Практический пример

Пользователь пишет:

“Нужен проект для удалённого периметра базы, 12 камер, 1 КПП, камеры частично на расстоянии больше 100 метров от серверной.”

Как проходит агент:

Parser
вытаскивает:
— объект: база
— камер: 12
— КПП: 1
— есть удалённые сегменты

Normalizer
превращает “больше 100 метров” в distance_risk = high

Planner
решает:
— топология не центральная по меди,
— нужен распределённый шкаф,
— uplink до удалённой точки — оптика

Selector
из БД выбирает:
— уличные камеры,
— уличный шкаф,
— PoE-коммутатор,
— SFP/оптический uplink

Builder
создаёт граф:
— серверная → оптика → удалённый шкаф
— шкаф → медь → камеры

Validator
проверяет:
— медные сегменты от шкафа до камер в пределах нормы
— PoE хватает
— портов хватает

Presenter
выдаёт:
— черновик архитектуры
— список оборудования
— предупреждения
— вопросы, где данных не хватает

Вот это уже выглядит как реальный инженерный агент.

Одна очень важная поправка

Не называй этот слой “агент, который проектирует систему”.
Это слишком широко и опасно.

Правильнее мыслить так:

“Агент интерпретирует задачу и запускает проектный конвейер.”

Почему это важно:
потому что проектирует у тебя не магия, а связка:

— LLM,
— БД оборудования,
— шаблоны,
— графовая модель,
— валидатор правил.

Это делает систему устойчивой.

Краткий вывод

Архитектура твоего ИИ-агента должна быть конвейерной.

Не один “умный бот”, а несколько ролей:

— понять задачу,
— нормализовать требования,
— выбрать типовую топологию,
— подобрать оборудование из БД,
— собрать граф проекта,
— проверить правила,
— объяснить результат человеку.

Главная практическая идея:
LLM должен понимать и объяснять, а код должен считать и проверять.

Раздел 2. База оборудования: не склад названий, а рабочая память инженера

В реальной жизни база оборудования нужна затем, чтобы агент не “придумывал”, а опирался на твой реальный набор решений. Без этого он будет вести себя как продавец-фантазёр: красиво говорить, но подсовывать случайные модели, не учитывать привычные связки, забывать про лицензии, PoE, форм-фактор, среду установки и твои рабочие предпочтения.

Поэтому база оборудования в твоём проекте — это не просто каталог. Это инженерная память системы.

Простым языком:

Без базы:
"Наверное сюда подойдет какой-нибудь 24-портовый switch"

С базой:
"Для этого узла из твоего набора подходят SW-24-POE-A и SW-24-POE-B,
но у A хватает PoE-бюджета под 18 камер, а у B нет"

Вот в этом и разница между игрушкой и инструментом.

Самая частая ошибка здесь такая: сделать таблицу вида:

vendor | model | category

и думать, что этого достаточно.

Этого недостаточно.

Почему? Потому что инженер подбирает не название, а свойства.
Система должна уметь ответить не только на вопрос “что это за устройство?”, но и на вопросы:

— где его можно применять,
— с чем оно совместимо,
— какие у него ограничения,
— какую роль оно играет в схеме,
— можно ли им закрыть конкретную задачу.

Поэтому правильнее мыслить не “списком моделей”, а несколькими слоями данных.

Схема:

[Справочник устройств]
        +
[Технические характеристики]
        +
[Инженерные роли]
        +
[Совместимость]
        +
[Коммерческие и рабочие статусы]

Теперь разложим это по-человечески.

Первый слой — что это вообще за сущность

Это базовая карточка устройства.

Например:

ID: CAM-HIK-4MP-BULLET-01
Category: camera
Subcategory: bullet
Vendor: Hikvision
Model: DS-2CD-XXXX
Title: 4MP Bullet Camera
Status: active

Этот слой нужен, чтобы устройство вообще существовало в системе.

Но сам по себе он почти бесполезен. Настоящая ценность — во втором слое.

Второй слой — технические параметры

Вот здесь устройство перестаёт быть названием и становится инженерным объектом.

Для камеры это могут быть:

— разрешение,
— средний битрейт,
— PoE / питание,
— уличное или внутреннее исполнение,
— поддержка аналитики,
— тип объектива,
— тепловизионная или обычная,
— совместимость с VMS / ONVIF.

Для коммутатора:

— число портов,
— число PoE-портов,
— PoE budget,
— uplink SFP,
— пропускная способность,
— монтаж в шкаф / DIN / настольный,
— рабочая температура.

Для сервера:

— тип роли,
— канальная ёмкость,
— допустимое число камер,
— дисковая подсистема,
— поддерживаемые сервисы.

То есть у разных категорий параметры разные.

И вот здесь появляется важное архитектурное решение.

Нельзя запихивать все параметры всех устройств в одну толстую таблицу с сотней колонок.

Плохой путь:

devices
- id
- vendor
- model
- resolution
- bitrate
- poe_budget
- sfp_ports
- relay_count
- storage_tb
- face_recognition
- ...

Почему плохо:
— половина полей всегда пустая,
— тяжело расширять,
— неудобно валидировать,
— категории устройств слишком разные.

Правильнее делать так:

devices                 // общее
device_specs_generic    // общие параметры
device_specs_camera     // только камеры
device_specs_switch     // только коммутаторы
device_specs_server     // только серверы
device_specs_controller // только СКУД

То есть есть общий “паспорт”, а дальше — профиль по типу устройства.

Это как в реальной жизни:
у всех есть паспорт, но у грузовика и тепловизора разные технические характеристики.

Таблица-схема

Слой	Что хранит	Пример
devices	общая карточка	vendor, model, category
category_specs	техпараметры по типу	bitrate, poe_budget, sfp_ports
ports	реальные интерфейсы	RJ45, SFP, RS-485, relay
capabilities	функции	ANPR, face, thermal
compatibility	правила совместимости	камера → VMS, терминал → контроллер
commercial	рабочий статус	active, deprecated, preferred

Теперь третий слой — инженерная роль устройства

Это очень полезная вещь, о которой часто забывают.

Одно дело “что это такое”, другое — “для чего это обычно используется”.

Например, у тебя могут быть две похожие камеры, но одна чаще идёт:
— на периметр,
а другая:
— в офис или коридор.

Или два коммутатора:
— один удобен как шкафной access switch,
— другой — как uplink/core.

Поэтому устройству полезно хранить инженерные роли:

roles:
- perimeter_camera
- indoor_camera
- checkpoint_anpr
- access_switch
- remote_outdoor_switch
- central_recording_server

Это нужно агенту, чтобы он подбирал оборудование не только по “сухим цифрам”, но и по практическому смыслу.

Четвёртый слой — возможности и функции

Это отвечает на вопрос “что устройство умеет”.

Например:

capabilities:
- poe
- onvif
- thermal
- face_recognition
- anpr
- outdoor
- rackmount
- redundant_psu

Почему это полезно:
если пользователь написал “нужно распознавание номеров”, агент не должен лезть в bit rate и разбираться по косвенным признакам. Он должен быстро найти устройства с capability = anpr.

То есть capabilities — это быстрый смысловой фильтр.

Пятый слой — совместимость

Вот это уже самый инженерный уровень.

Потому что “подходит по описанию” ещё не значит “стыкуется в системе”.

Примеры совместимости:

— камера поддерживается нужной VMS,
— терминал работает с конкретным контроллером,
— коммутатор даёт нужный PoE-класс,
— ANPR-камера требует определённый модуль / лицензию,
— тепловизор требует отдельной логики отображения.

Это лучше хранить не “в голове агента”, а отдельно как правила.

Например:

compatibility_rules
- source_device_id
- target_device_id
- rule_type
- status
- note

или более гибко:

compatibility_rules
- source_category
- target_category
- condition
- result

Пример:

source_category = anpr_camera
target_category = vms
condition = requires_anpr_module = true
result = compatible_with_license

Теперь шестой слой — рабочий статус

Это слой “жизненности”.

В реальной работе тебе важно не только, что устройство технически годится, но и:

— используешь ли ты его сейчас,
— снято ли оно с поставок,
— рекомендованное ли оно,
— есть ли более удачная замена,
— хочешь ли ты вообще, чтобы агент его предлагал.

Поэтому нужны статусы вроде:

active
preferred
legacy
deprecated
blocked
experimental

Это мелочь, но она сильно улучшает поведение системы.

Например:
модель в базе есть, но deprecated, и агент может показывать её только если нет лучшей альтернативы.

Практический пример

Допустим, агенту нужно подобрать узел на 16 уличных камер.

Если у него плохая база, он увидит просто список коммутаторов и выберет любой 24-портовый.

Если база хорошая, он проверит:

— категория: switch
— роль: access_switch / outdoor_node
— PoE ports ≥ 16
— PoE budget достаточный
— uplink: SFP есть
— environment: подходит для размещения в нужном шкафу
— status: preferred/active

И тогда выбор становится осмысленным.

ASCII-схема хорошей карточки устройства

[Device]
  id: SW-24-POE-01
  category: switch
  subcategory: poe_access
  vendor: Hikvision
  model: XXX
  status: preferred

[Role]
  - access_switch
  - camera_node

[Capabilities]
  - poe
  - sfp_uplink
  - rackmount

[Specs]
  - ports_rj45: 24
  - ports_poe: 24
  - poe_budget_w: 370
  - sfp_ports: 2

[Ports]
  - GE1..GE24
  - SFP1..SFP2
  - PWR

Вот такая карточка уже полезна агенту.

Теперь главный практический вопрос:
как проектировать БД на старте, чтобы не утонуть?

Ответ: не пытаться описать всё идеально с первого дня.

Для MVP тебе достаточно 6 основных сущностей:

devices — общая карточка

device_ports — реальные порты

device_capabilities — функции и теги

device_roles — инженерные роли

category_specs_* — техпараметры по категориям

compatibility_rules — ключевые правила совместимости

То есть стартовая схема может быть такой:

devices
device_ports
device_roles
device_capabilities
camera_specs
switch_specs
server_specs
controller_specs
compatibility_rules

Это уже достаточно серьёзно, но ещё не монстр.

Одна важная поправка

Ты можешь захотеть хранить “всё” в JSON, потому что это быстро.

Это частично хорошая идея, но только частично.

Правильный компромисс такой:

— критичные поля для фильтрации и расчётов — отдельными колонками,
— редкие и нестандартные параметры — в JSON.

Например:

Хорошо вынести отдельно:
— category
— vendor
— model
— status
— ports_count
— poe_budget
— bitrate
— storage_class
— environment

А вот экзотику можно держать в JSON:
— спецрежимы,
— редкие аналитики,
— дополнительные замечания.

То есть:

важное для логики -> columns
редкое и вспомогательное -> jsonb

Это очень практичный путь для PostgreSQL.

Краткий вывод

База оборудования в твоём проекте — это не каталог названий, а инженерная память системы.

Хорошая база должна хранить не только:
— что это за устройство,

но и:
— какие у него параметры,
— какие у него порты,
— какую роль оно играет,
— что оно умеет,
— с чем оно совместимо,
— можно ли его вообще сейчас рекомендовать.

Если упростить до одной фразы:

агент должен выбирать не “модель из списка”, а “подходящий инженерный узел из рабочего набора”.

Раздел 3. Движок инженерных проверок: система должна уметь говорить «так нельзя»

В реальной работе проектировщик постоянно выполняет один и тот же тип мыслительной операции: он проверяет, не нарушает ли предложенное решение физические, сетевые и нормативные ограничения.
Причём делает это почти автоматически:

— хватит ли портов
— хватит ли PoE
— не слишком ли длинный кабель
— выдержит ли сеть поток
— хватит ли дискового пространства
— совместимы ли устройства

Если конструктор архитектуры не умеет проверять такие вещи, он превращается в красивый редактор схем.

Поэтому в твоей системе должен существовать отдельный компонент — validation engine (движок инженерных проверок).

Его задача — брать построенную архитектуру и отвечать на вопрос:

Можно ли физически и логически собрать такую систему?

Если нет — он должен объяснить почему и предложить варианты исправления.

1. Общая идея движка проверок

Система уже имеет модель проекта:

Project
 ├─ Devices
 ├─ Ports
 └─ Links

Validator проходит по этой структуре и проверяет правила.

Схема:

Project Graph
      │
      ▼
Validation Engine
      │
      ▼
Issues
Warnings
Recommendations
2. Типы проверок

Все инженерные проверки можно разделить на несколько классов.

Тип	Что проверяется
Interface	совместимость портов
Capacity	порты, PoE, пропускная способность
Physical	длина кабеля
Network	топология сети
Storage	ёмкость архива
Compatibility	совместимость устройств
3. Проверка интерфейсов

Самая базовая проверка — можно ли соединять эти два порта.

Пример:

Camera.RJ45 -> Switch.RJ45

это корректно.

Но:

Camera.RJ45 -> Controller.RS485

невозможно.

Правило проверки:

if portA.interface != portB.interface
   error

Иногда допускаются адаптеры, но это отдельная логика.

4. Проверка портов

Коммутатор не может обслужить бесконечное количество устройств.

Например:

Switch
24 ports

Если подключено:

27 камер

это ошибка.

Алгоритм:

for each device
   count active links per port
   if > max_connections
      error
5. Проверка PoE бюджета

PoE — одна из самых частых реальных проблем.

Коммутатор может иметь:

PoE budget = 370W

а камеры потребляют:

16W × 24 = 384W

Это уже превышение.

Формула:

total_power = sum(device.power_consumption)

if total_power > switch.poe_budget
   error
6. Проверка длины кабеля

Это твоя ключевая инженерная фишка.

Для медного Ethernet:

channel_length ≤ 100m
recommended ≤ 90m

Validator проверяет:

if link.calculated_length > cable.max_length
   error

Также можно делать градацию:

80-90m → warning
>100m → error
7. Проверка пропускной способности

Каждая камера создаёт поток.

Например:

camera bitrate = 6 Mbps

16 камер:

96 Mbps

Если uplink:

Fast Ethernet (100 Mbps)

это почти перегрузка.

Validator считает:

sum(camera_bitrate) < link.bandwidth

Если нет — предупреждение.

8. Проверка архива

Для видеосистемы важно хранение.

Формула приблизительного расчёта:

storage =
bitrate × cameras × seconds × days

Пример:

6 Mbps × 24 cameras × 86400 × 30
≈ 466 TB raw

Но учитываются:

— кодек
— движение
— VBR
— аналитика

Validator может делать упрощённую оценку.

9. Проверка топологии

Иногда архитектура может быть логически невозможной.

Например:

camera -> camera

или

switch -> switch
(no uplink to server)

Validator должен искать:

— изолированные сегменты
— циклы
— отсутствие центрального узла

Это делается через анализ графа.

10. Проверка совместимости устройств

Пример:

Face Terminal -> Controller

Но терминал может поддерживать только:

Wiegand

а контроллер:

RS485

Это несовместимо.

Validator проверяет через таблицу правил.

11. Структура результата проверки

Результат должен быть структурированным.

Пример:

{
 "status": "warning",
 "issues": [
  {
   "type": "poe_budget",
   "severity": "error",
   "device": "SW-01",
   "message": "PoE budget exceeded"
  },
  {
   "type": "link_length",
   "severity": "warning",
   "link": "L-24",
   "message": "Copper link length near limit"
  }
 ]
}
12. Система уровней серьёзности

Полезно иметь уровни.

Level	Значение
info	просто замечание
warning	риск
error	невозможно реализовать
13. Предложения по исправлению

Validator не должен только ругаться.

Он должен давать варианты.

Пример:

Ошибка:
PoE budget exceeded

Решения:
1 добавить второй PoE switch
2 использовать 48-port switch
3 разделить камеры на два узла
14. Архитектура валидатора

Лучше делать его как модуль правил.

ValidationEngine
   │
   ├─ PortRules
   ├─ CableRules
   ├─ NetworkRules
   ├─ PowerRules
   └─ CompatibilityRules

Каждое правило — отдельная функция.

15. Порядок проверки

Правильный порядок:

1 ports
2 interface
3 cable length
4 power
5 bandwidth
6 storage
7 topology

Это ускоряет диагностику.

16. Почему это важнейшая часть проекта

ИИ может:

— понять задачу
— подобрать оборудование
— предложить архитектуру

Но только validator может сказать:

это реально можно построить

Именно поэтому этот компонент превращает проект из AI-игрушки в инженерный инструмент.

17. Практический пример

Пользователь описывает объект:

24 камеры
1 коммутатор
расстояние до части камер 120 м

Агент предлагает:

24 камеры -> switch

Validator проверяет:

port_count = OK
poe_budget = OK
cable_length = ERROR

И выдаёт:

Рекомендуемая архитектура:

server
   │
 fiber
   │
outdoor switch
   │
 copper
   │
cameras

То есть он исправляет архитектуру.

Краткий вывод

Validator — это сердце инженерной системы.

Он:

— проверяет физические ограничения
— контролирует сеть
— анализирует питание
— проверяет нормативы
— обнаруживает логические ошибки

И самое важное — объясняет инженеру почему решение плохое и как его исправить.

Часть 3. Архитектура всей системы
Глава 1. Как собрать все компоненты в одно приложение
Раздел 1. Общая архитектура платформы: как соединяются UI, ИИ-агент, база оборудования и движок проверок

В реальной жизни сложные инженерные системы почти никогда не строятся как одно большое приложение. Если всё смешать в один код — интерфейс, ИИ, базу оборудования, проверки, расчёты — то через несколько месяцев система превращается в хаос. Любое изменение ломает что-то другое.

Поэтому взрослые системы проектируются как несколько независимых модулей, каждый из которых делает одну работу.

Твой проект — хороший пример системы, где это важно.

У тебя есть:

— интерфейс конструктора
— база оборудования
— граф архитектуры
— AI-агент
— движок инженерных проверок

Если их не разделить, всё быстро станет неуправляемым.

Правильная мысль:
система должна быть модульной платформой проектирования.

1. Основные модули системы

В самом простом виде архитектура выглядит так:

UI
 │
 ▼
API
 │
 ├── Project Engine
 │
 ├── Equipment Database
 │
 ├── Validation Engine
 │
 └── AI Agent

Каждый из этих блоков имеет свою роль.

Модуль	Задача
UI	визуальный конструктор архитектуры
API	центральная точка логики
Project Engine	хранит граф системы
Equipment DB	база устройств
Validation Engine	проверка инженерных правил
AI Agent	интерпретация задач
2. Project Engine — ядро системы

Это самый важный компонент.

Он хранит архитектуру проекта.

Не текст.
Не картинку.
А структуру системы.

Например:

Project
 ├─ Devices

Глава 2. Пользовательский интерфейс конструктора
Раздел 2. Архитектура UI-конструктора: как человек будет реально проектировать систему

В реальной жизни инженер не проектирует систему через формы и таблицы. Он почти всегда думает схемой.

Когда человек проектирует систему безопасности, он мысленно рисует примерно такую картину:

Camera ── Switch ── Core ── Server
          │
          └── Cabinet

То есть мышление происходит графом объектов.

Поэтому интерфейс такого инструмента почти всегда строится как node-editor — визуальный редактор узлов и связей.

Если сделать интерфейс через формы (“добавить камеру”, “добавить коммутатор”, “выбрать кабель”), работать будет крайне неудобно.

Поэтому правильная модель интерфейса — инженерный canvas.

1. Основная идея интерфейса

Пользователь работает с визуальной схемой.

        [Camera]
           │
           │ Cat6
           │
[Camera]──[Switch]──[Server]
           │
           │ Fiber
           │
        [Core]

Каждый элемент — это узел (node).

Каждое соединение — link.

Это визуальное представление той же модели:

Device
Port
Link
2. Что пользователь должен уметь делать

UI должен позволять делать четыре базовых действия.

Действие	Что происходит
добавить устройство	создаётся Device
удалить устройство	удаляется Device
соединить порты	создаётся Link
изменить параметры	обновляется модель

То есть UI — это просто инструмент редактирования графа.

3. Визуальная модель устройства

Раздел 3. Связи, трассы и физика кабеля: как превратить линию на схеме в реальный инженерный объект

В реальной жизни кабель — это почти половина проекта системы безопасности. Камеры, коммутаторы и серверы занимают внимание, но именно кабельная инфраструктура чаще всего становится источником проблем: превышение длины, неправильный тип кабеля, плохая прокладка, отсутствие резерва.

Поэтому в твоей системе линия связи не должна быть просто “чёрной линией между двумя устройствами”. Она должна быть полноценным инженерным объектом, который описывает реальную физическую трассу.

Простым языком:

Плохая модель:
Camera ---- Switch

Хорошая модель:
Camera --Link-- Route -- Cable -- Switch

То есть соединение должно иметь несколько уровней описания.

1. Три уровня описания линии

Любая связь в системе должна состоять из трёх частей.

Уровень	Что описывает
Link	логическое соединение
Route	маршрут прокладки
Cable	тип кабеля

Схема:

Device ─ Port ─ Link ─ Route ─ Cable ─ Device

Это может показаться сложным, но именно так думают инженеры.

2. Link — логическая связь

Link говорит только о том, что два порта соединены.

Пример:

Camera.RJ45 -> Switch.Port12

Структура:

Link
 id
 from_port
 to_port
 cable_id
 route_id
 bandwidth_required
 poe_required

Он не хранит всю физику линии.

3. Cable — тип кабеля

Cable описывает свойства кабеля.

Пример:

Cable
 id
 name
 category
 max_length
 bandwidth
 poe_supported
 environment

Примеры типов:

Cat5e
Cat6
Fiber SM
Fiber MM
RS485
Coax

Каждый имеет свои ограничения.

4. Route — маршрут трассы

Вот здесь появляется реальная физика.

Route описывает как кабель проходит по объекту.

Например:

Route
 id
 route_length
 vertical_length
 termination_reserve
 service_loop
 calculated_length
 environment

Формула:

calculated_length =
route_length
+ vertical_length
+ termination_reserve
+ service_loop
5. Пример расчёта

Инженер говорит:

“камера примерно в 70 метрах”.

Но трасса на самом деле:

60 м по фасаду
6 м подъём
4 м запас

Результат:

calculated_length = 70

Validator работает уже с этой величиной.

6. Типы среды прокладки

Очень важно учитывать среду.

environment
 indoor
 outdoor
 underground
 industrial
 high_em_noise

Это влияет на выбор кабеля.

Например:

— в промышленной зоне может быть предпочтительна оптика
— на улице требуется наружная оболочка

7. Уровни длины линии

<80m	безопасно
80-90m	предупреждение
90-100m	риск

100m | ошибка |

Например:

Camera

 │ Cat6 30m
 │

 │
 │ Fiber 120m
 │
Server

Тогда route состоит из сегментов.

Это полезно для больших объектов.
